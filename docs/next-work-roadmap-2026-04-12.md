# DevFlow 다음 작업 로드맵

작성일: 2026-04-12

## 목적

현재 MVP가 "기록 -> 집계 -> 리포트" 흐름까지 연결된 상태이므로, 다음 작업은 기능 확장과 제품 고도화를 함께 관리한다.
이 문서는 당장 구현할 TODO보다 넓은 백로그이며, 우선순위와 완료 기준을 잡기 위한 기준 문서다.

## 우선순위 요약

1. Settings 확장과 타이머 기본값 연결
2. 카테고리 관리 기능
3. 세션 기록 편집/삭제
4. 리포트와 인사이트 고도화
5. GA4/GTM 운영 분석 연결
6. UX 품질과 접근성 개선
7. 데이터 안정성, 성능, 운영 문서 정리

## 1. Settings 확장

목표:
사용자가 자신의 집중 루틴을 저장하고, 타이머 시작 화면에서 바로 이어 쓸 수 있게 만든다.

작업 후보:

- 기본 집중 시간 설정
- 기본 휴식 시간 설정
- 하루 목표 집중 시간과 목표 세션 수 분리 저장
- 타이머 시작 폼에 저장된 기본값 자동 반영
- 알림 on/off 설정 구조 추가
- settings 저장 후 대시보드와 타이머의 데이터 재검증 흐름 정리

완료 기준:

- `/settings`에서 저장한 기본 집중/휴식 시간이 `/timer`에 반영된다.
- 목표 관련 값과 타이머 preset 값의 역할이 사용자에게 분명히 구분된다.

## 2. 카테고리 관리

목표:
기본 카테고리만 쓰는 상태에서 벗어나, 사용자별 작업 분류를 직접 관리할 수 있게 만든다.

현재 상태:

- 사용자 커스텀 카테고리 생성 최소 버전 완료
- 사용자 커스텀 카테고리 이름/색상 수정 완료
- 카테고리 생성/수정 실패 토스트 표시 완료
- Settings에서 기본 카테고리와 사용자 카테고리 목록 표시 완료

작업 후보:

- 카테고리 비활성화 또는 삭제 정책 정의
- 타이머, 히스토리, 주간 리포트에 커스텀 카테고리 반영

완료 기준:

- 사용자가 직접 만든 카테고리를 세션 저장 시 선택할 수 있다.
- 히스토리와 리포트에서 커스텀 카테고리가 기본 카테고리와 같은 기준으로 집계된다.

## 3. 세션 기록 관리

목표:
잘못 저장된 기록을 수정할 수 있게 해서 실제 사용 가능성을 높인다.

작업 후보:

- 완료 세션 상세 보기
- 세션 기록 수정
- 세션 삭제 또는 취소 처리
- 시작/종료 시간 수정 범위 정의
- 세션 메모 수정
- 카테고리, 난이도, 집중도 수정
- 수정/삭제 후 대시보드, 히스토리, 리포트 재검증

완료 기준:

- 사용자가 실수로 저장한 세션을 직접 정리할 수 있다.
- 수정 이후 모든 집계 화면의 값이 일관되게 갱신된다.

## 4. 타이머 경험 고도화

목표:
타이머가 단순 입력 화면이 아니라 실제 집중 세션을 믿고 켜둘 수 있는 도구가 되게 한다.

작업 후보:

- 브라우저 탭 이탈 후에도 진행 시간 계산 안정화
- 새로고침 후 active 세션 복원 UX 개선
- 집중 시간 초과 시 안내 문구 정리
- 휴식 타이머 또는 다음 세션 제안
- 중단 사유 입력
- 세션 종료 후 기록 입력 단계의 흐름 축소
- 모바일에서 타이머 컨트롤 영역 고정 여부 검토

완료 기준:

- 사용자가 타이머를 켜고 이동/새로고침해도 세션 상태를 이해할 수 있다.
- 세션 시작, 중단, 재개, 완료 흐름이 한 화면에서 자연스럽게 이어진다.

## 5. 대시보드 고도화

목표:
오늘 상태를 빠르게 판단할 수 있는 요약 화면으로 만든다.

작업 후보:

- 오늘 남은 목표 시간 계산
- 최근 7일 대비 오늘 진행률 비교
- 가장 집중이 잘 된 시간대 표시
- 오늘의 카테고리 분포 표시
- 중단 세션이 많을 때 짧은 경고 문구 표시
- 빈 상태와 첫 세션 안내 개선

완료 기준:

- 대시보드만 보고 오늘 더 집중해야 하는지, 이미 목표에 가까운지 판단할 수 있다.

## 6. 주간 리포트와 인사이트 고도화

목표:
단순 통계 나열이 아니라 다음 행동을 정하는 리포트로 발전시킨다.

작업 후보:

- 완주율, 중단률, 평균 집중 길이 규칙 세분화
- 카테고리 편중 분석 강화
- 평일/주말 패턴 구분
- 시간대별 집중 품질 분석
- 주간 요약 문장 품질 개선
- 지난주 대비 변화량 추가
- 리포트용 차트 시각화 검토

완료 기준:

- 사용자가 "다음 주에 무엇을 바꿀지"를 리포트에서 바로 얻을 수 있다.

## 7. 분석 이벤트와 운영 지표

목표:
제품 사용 흐름을 최소한으로 추적해 개선 우선순위를 판단할 수 있게 한다.

작업 후보:

- GA4 측정 ID 확인
- GTM에서 GA4 Google 태그 게시
- page_view 수집 확인
- 핵심 이벤트 정의
- 회원가입 시작 이벤트
- 로그인 성공 이벤트
- 타이머 시작 이벤트
- 세션 완료 이벤트
- 설정 저장 이벤트
- 주간 리포트 조회 이벤트
- 자유 입력값을 이벤트 파라미터로 보내지 않는 기준 정리

완료 기준:

- GA4 실시간 보고서에서 페이지뷰가 잡힌다.
- 핵심 전환 이벤트가 개인정보를 포함하지 않는 형태로 수집된다.

## 8. UX 품질 개선

목표:
MVP가 "작동한다"에서 "계속 쓰고 싶다"로 넘어가도록 화면 완성도를 높인다.

작업 후보:

- 모바일 레이아웃 재점검
- 폼 에러 메시지 톤 통일
- 로딩 상태 통일
- 빈 상태 문구 개선
- 저장/삭제/수정 토스트 통일
- 버튼 라벨과 액션 의미 재점검
- 다크 모드에서 select, input, card 대비 재점검
- 키보드 접근성과 focus 상태 점검

완료 기준:

- 주요 화면에서 사용자가 다음 행동을 헷갈리지 않는다.
- 모바일에서도 타이머와 기록 입력이 답답하지 않다.

## 9. 데이터와 보안 안정성

목표:
사용자별 데이터 분리와 변경 흐름을 더 단단하게 만든다.

작업 후보:

- RLS 정책 재점검
- 사용자 카테고리 삭제 시 기존 세션 처리 정책 정의
- 세션 수정/삭제 서버 액션에서 소유자 검증
- daily_goals upsert 정책 재점검
- Supabase 타입 갱신 절차 문서화
- 테스트용 데이터와 운영 데이터 분리 기준 정리

완료 기준:

- 사용자는 본인 데이터만 조회/수정할 수 있다.
- 데이터 수정 기능이 추가되어도 기존 리포트 집계가 깨지지 않는다.

## 10. 성능과 배포 품질

목표:
배포 후에도 안정적으로 관리할 수 있는 구조를 만든다.

작업 후보:

- 중복 인증 조회 지점 재확인
- 서버 컴포넌트 쿼리 재사용 범위 검토
- `revalidatePath()` 호출 범위 정리
- 동적 렌더링 유지가 필요한 라우트 문서화
- Vercel 환경 변수와 로컬 `.env` 역할 분리
- Production 배포 체크리스트 작성
- `NEXT_PUBLIC_SITE_URL` Production 값 확인

완료 기준:

- 배포 후 SEO 메타데이터와 sitemap 기준 URL이 일관된다.
- 데이터 변경 후 필요한 화면만 갱신된다.

## 추천 작업 순서

### 단기

1. 카테고리 삭제/비활성화 정책
2. Settings 기본 집중/휴식 시간 연결
3. 세션 기록 수정 최소 버전
4. GA4 측정 ID 연결 마무리

### 중기

1. 카테고리 수정/비활성화 정책
2. 주간 리포트 지난주 대비 지표
3. 대시보드 오늘 상태 요약 강화
4. 핵심 이벤트 추적

### 후반

1. 차트 시각화 개선
2. 접근성/모바일 품질 점검
3. 데이터 정책 문서화
4. 배포 체크리스트와 운영 문서 정리

## 메모

- 당장 사용자 체감이 큰 것은 Settings, 카테고리, 세션 수정이다.
- 운영 지표는 GA4 연결이 끝난 뒤 이벤트 이름과 파라미터를 작게 시작하는 편이 좋다.
- 인사이트 고도화는 입력 데이터가 늘수록 품질이 좋아지므로, 카테고리와 중단 사유 입력 이후에 확장하면 효율적이다.
- 실제 다음 작업 큐는 `docs/remaining-work-queue-2026-04-12.md`에서 관리한다.
